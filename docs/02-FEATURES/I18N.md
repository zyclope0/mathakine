# Guide i18n - Systeme de traduction Mathakine

**Updated** : 2026-04-16  
**Framework** : `next-intl` `^4.9.1`  
**Frontend runtime** : Next.js 16 App Router

---

## Objectif

Documenter la verite i18n actuelle du frontend :

- messages FR/EN dans `frontend/messages/`
- selection de locale via Zustand
- rendu via `next-intl`
- scripts de verification dans `frontend/scripts/i18n/`

---

## Architecture

### Stack

- `next-intl` `^4.9.1`
- Zustand persiste pour la preference locale
- fichiers de messages JSON :
  - `frontend/messages/fr.json`
  - `frontend/messages/en.json`

### Flux runtime

1. la locale active est lue depuis `localeStore`
2. `frontend/components/providers/NextIntlProvider.tsx` charge les messages FR et EN par imports statiques
3. le provider selectionne le dictionnaire correspondant
4. `LocaleInitializer` resynchronise la locale et `document.documentElement.lang`

### Provider

Source de verite :

- `frontend/components/providers/NextIntlProvider.tsx`

Etat actuel :

- FR et EN sont tous deux charges via imports statiques
- il n'y a plus de chargement FR synchrone / EN asynchrone
- fallback de messages limite au dev via `console.warn`

## Utilisation dans le code

```tsx
import { useTranslations } from "next-intl";

export function Example() {
  const t = useTranslations("dashboard");

  return <h1>{t("title")}</h1>;
}
```

Principes :

- pas de copy inline pour une surface i18nisee
- garder les labels proches du domaine
- eviter les constantes top-level avec texte fige si elles alimentent directement le rendu

## Scripts de verification

Depuis `frontend/` :

```bash
npm run i18n:validate
npm run i18n:check
```

Audit global optionnel :

```bash
npm run i18n:extract
# ou
npm run i18n:all
```

Important :

- `validate` + `check` sont la verification minimale fiable pour un lot i18n cible
- `extract` / `all` sont plus larges et peuvent encore remonter des hardcodes hors du lot traite

## Etat terrain au 2026-04-10

Ce qui est ferme :

- surfaces utilisateur principales i18nisees
- route handlers chat / stream relies a `fr.json` / `en.json`
- pages route-level admin et `offline` sorties des hardcodes directs
- images sociales et pages systeme globales coherentes avec les messages

Dette i18n residuelle connue :

- `npm run i18n:extract` peut encore signaler des hardcodes hors des lots i18n fermes

---

## Bonnes pratiques

1. ajouter la cle dans `fr.json` et `en.json` dans le meme namespace
2. utiliser `useTranslations(...)` dans le composant ou hook adapte
3. verifier `npm run i18n:validate` et `npm run i18n:check`
4. ajouter un test si le lot touche une surface critique ou une page entiere
5. ne pas presenter `i18n:all` comme un gate local strict si le lot est scope et que l'extract remonte du bruit hors perimetre
