## Verdict global

**Le projet n'est PAS un cas sévère de DRY.** Pour un projet de 120 K LOC développé seul en 4 mois IA-assisté, le niveau de duplication est faible. L'architecture imposée (séparation `services/`, `handlers/`, `routes/`, hooks dédiés, `utils/` partagés) a bien joué son rôle de barrière naturelle contre la prolifération du code dupliqué.

Les violations identifiées sont :
- **localisées** — dans des zones précises, pas systémiques
- **typiques de la vitesse** — features parallèles (exercices vs défis) évoluées séparément sans unification systématique
- **sans impact fonctionnel immédiat** — aucune ne produit de bug aujourd'hui

Score DRY estimé : **8/10** (violations mineures à moyennes, pas de monolithe copié-collé)

## Ce qui n'est PAS du DRY (structure saine)

| Pattern | Pourquoi ce n'est pas du DRY |
|---------|------------------------------|
| `challenge_service.py` vs `exercise_attempt_service.py` | Domaines métier distincts, interfaces différentes |
| Hooks `useExercises` vs `useChallenges` | Wrappent des endpoints différents, abstraction légitime |
| `challenge_ai_service.py` vs `exercise_ai_service.py` | Contrats SSE différents (IA9 response_mode côté défis) |
| `exercise_attempt_service` vs `challenge_attempt_service` | Logiques de tentative distinctes |
| `build_authenticated_user_payload` vs `get_current_user_payload` | Deux fonctions d'auth session avec des responsabilités distinctes (création vs validation) — la duplication des champs est le vrai problème, pas les fonctions elles-mêmes |
