# Mathakine — contexte design (teach-impeccable)

Ce fichier est la **source de vérité** UI/UX pour Mathakine. À mettre à jour quand la direction marque ou les contraintes d’accessibilité évoluent. Une copie orientée outil est dans `.github/copilot-instructions.md` (utile aussi pour Cursor / Claude / autres assistants).

---

## Design Context

### Users

- **Apprenants (enfants / collégiens)** : résoudre des exercices et défis, progression, badges, feedback clair. L’exercice reste le centre d’attention ; charge cognitive **minimale**.
- **Adultes (parents / enseignants)** : suivi, classes, surfaces payantes — lisibilité et efficacité, **sans** recopier les patterns “dashboard SaaS générique”.
- **Contexte** : plateforme **multilingue (fr/en)** ; web (Next.js) ; patterns **maintenables** (solo founder).

### Brand Personality (3 mots — validé produit)

1. **Apaisant** — zéro surcharge sensorielle ; charge cognitive minimale.
2. **Évident** — navigation **tubulaire** ; pas de menus cachés ; principe **“boring but obvious”** (évidence avant originalité décorative).
3. **Constructif** — l’erreur fait partie de l’apprentissage ; feedback **doux et non punitif**.

### Aesthetic Direction

**Références (à suivre, pas à copier)**

| Source | Ce qu’on en retient |
|--------|---------------------|
| **Gov.uk** | Accessibilité **cognitive** et clarté absolue. |
| **Khan Academy** | Usage massif de **whitespace** et d’air. |
| **Duolingo** | Uniquement la **mécanique** streak / gamification — avec **~80 % de bruit visuel en moins** qu’eux. |

**Anti-références (à fuir absolument)**

- **Dashboards B2B SaaS** façon Vercel, Stripe, Linear (look “grid produit” froid).
- **“Grid of Boxes”** : cartes avec `border` + `shadow-sm` partout pour structurer l’écran.
- **Séparation** : elle se fait par **l’espace** (`padding` / `gap`), **pas** par des lignes ou des cadres systématiques.

### Signature visuelle — « Constellation de la progression »

- **Constellation** (nœuds reliés, forme géométrique sobre) : visualise l’**arc de progression persistant** (niveaux / paliers) — ancrage visuel **stable** et rassurant.
- **Série (streak)** : **indicateur séparé** (texte discret ou widget dédié), **sans** le même motif graphique que la constellation — éviter confusion et charge liée à la perte de série.
- Sur une **même page**, ne pas empiler **deux** motifs « constellation » sans rôle distinct (ex. arc persistant vs défis du jour) ; une métaphore, une lecture.

**Stack technique (référence implémentation)**

- Next.js 15, Tailwind CSS v4, tokens type shadcn, **Nunito** (UI), **JetBrains Mono** (code — `next/font`), Radix, variante `.dark`, KaTeX pour les maths, i18n fr/en.

### Thème clair / sombre

- **Les deux sont d’égale importance.** Le mode sombre est une **nécessité d’accessibilité** (hypersensibilité visuelle, confort).
- **Règle absolue (mode clair)** : **interdiction** d’utiliser le **blanc pur** (`#FFFFFF` / `bg-white`) comme fond de page — risque d’éblouissement / halo **délétère** pour les profils TSA / TDAH. Préférer des fonds **muted** (voir palette).

### Palette “Muted” (direction)

| Usage | Direction |
|--------|-----------|
| **Fond principal** | Gris-bleu très clair et doux — en prod le thème spatial par défaut utilise un fond teinté via tokens (ex. `#f0f4ff`), pas blanc pur ; équivalents sémantiques possibles (`slate-50` / `zinc-50`). |
| **Cartes / conteneurs** | Surfaces discrètes, **très peu** de contraste avec le fond ; **pas** d’ombres décoratives type grille SaaS — ombres **minimales** seulement si elles servent la hiérarchie ou l’état (focus, survol). |
| **Erreur / aide** | Jamais rouge vif (`bg-red-500`). Tons **terreux / doux** (ex. `rose-100` fond, `rose-800` texte). |
| **Succès** | Vert doux (ex. `emerald-100` / `emerald-800`). |

Aligner progressivement les tokens CSS (`globals.css`) sur ces intentions si l’écart est trop fort.

### Accessibilité et inclusion (cible)

- **WCAG AA strict au minimum** (contrastes typographiques en particulier).
- **COGA (cognitive accessibility)** : divulgation progressive (cacher la complexité), **pas d’autoplay**, support **strict** de `prefers-reduced-motion` — les animations de récompense doivent pouvoir être **désactivées instantanément**.
- Cibles tactiles **≥ 44×44 px** ; modes déjà présents dans le projet (texte large, contraste, dyslexie, etc.) à respecter dans les nouveaux écrans.

### Design Principles (décisions produit + dépôt)

1. **Charge cognitive d’abord** — rien ne rivalise avec la tâche d’apprentissage ; pas d’animations décoratives pendant la réflexion.
2. **Évidence > originalité décorative** — navigation prévisible ; pas de patterns “cachés pour faire joli”.
3. **Feedback constructif** — erreurs et aides **non punitives** ; états chargement / succès / erreur **distincts** ; gamification **subtile** (mécanique utile, pas de bruit visuel).
4. **Structure par l’espace** — séparation par **padding / gap**, pas par grille de cartes bordées + ombres.
5. **Muted & pas de blanc pur (clair)** — fonds doux ; sombre aussi important pour l’accessibilité.
6. **WCAG AA + COGA + reduced-motion** — pas d’autoplay ; animations récompense **coupables** si l’utilisateur a demandé moins de mouvement.
7. **Cohérence technique** — Tailwind / tokens ; logique dans hooks ; TS strict ; i18n pour le texte utilisateur.
8. **Constellation ≠ streak** — le motif constellation sert l’arc **persistant** ; le streak reste **hors** de ce motif (pas de double lecture graphique).
